# RouteX Tauri 重写实施方案

## 1. 结论

RouteX 适合迁移到 Tauri，但不适合按当前 Electron 架构逐文件平移。

推荐路线：

1. 保留现有 React 渲染层。
2. 先抽离桌面能力适配层，去掉前端对 `window.electron` 和 `window.api` 的直接依赖。
3. 用 Tauri command、event、window、tray 和 sidecar 重建桌面壳。
4. 分阶段补齐系统代理、托盘、悬浮窗、自动更新和平台权限能力。

当前阶段状态：

1. 截至 2026-04-10，阶段 0 / 1 / 2 已完成。
2. 阶段 4 已按开发交付口径完成，阶段 5 已完成首版更新链路接入。
3. 后续推进重点转为阶段 3 / 5 / 6 的实机回归、发布链收口和最终清理，不再把阶段 0 / 1 / 2 视为主要阻塞项。

不推荐路线：

1. 直接把 Electron 的 IPC 通道一比一改写成 Tauri command。
2. 第一阶段就追求与 Electron 版完全等功能。
3. 在未收敛能力边界前重做 UI 或大规模调整页面结构。

## 2. 当前项目现状

当前仓库是 Electron + React + TypeScript 桌面应用，主成本不在页面，而在桌面能力层。

关键事实：

1. Electron 主进程入口在 `src/main/index.ts`，负责窗口生命周期、启动流程、托盘、快捷键、浮窗、内核启动和关停。
2. 预加载桥在 `src/preload/index.ts`，前端通过 `window.electron` 和 `window.api` 访问桌面能力。
3. React 主应用入口在 `src/renderer/src/main.tsx`，渲染层整体可复用性较高。
4. Mihomo 生命周期管理集中在 `src/main/core/manager.ts`，这里是迁移风险最高的模块之一。
5. 托盘逻辑集中在 `src/main/resolve/tray.ts`，包含动态菜单、状态联动和平台差异处理。
6. 系统代理逻辑集中在 `src/main/sys/sysproxy.ts`，同时依赖外部 service 能力。
7. 当前前后桥接层的调用面较大，粗略统计共有 163 个 invoke/send/on 通道，其中系统类通道最多。

这意味着：

1. React 页面、路由、状态层、组件层可以保留大部分。
2. Electron 主进程代码不能原样复用。
3. Tauri 重写的核心工作是重建能力边界，而不是重写页面。

## 3. 重写目标

### 3.1 必须达成

1. 保持当前主界面、主要页面和核心交互基本不倒退。
2. 能启动、停止、重启 Mihomo，并读取基础运行状态。
3. 能读写应用配置、订阅配置和运行时文件。
4. 能在 Windows、macOS、Linux 上完成基础打包和运行。
5. 前端不再依赖 Electron preload 全局对象。

### 3.3 文档同步要求

1. 每完成一个已验证的迁移项，都要同步更新 `docs/tauri-migration-status.md`。
2. 文档更新必须反映“完成内容、验证结果、剩余风险”三件事，不能只写结论。
3. 未验证通过的事项不能记为完成，只能写当前进展、阻塞或待验证状态。

### 3.2 首期不强求

1. 自定义托盘菜单的全部细节。
2. 悬浮窗的完整交互表现。
3. 自动更新链路一次到位。
4. 所有 Electron IPC 名称和调用方式保持不变。
5. 所有平台特例在首期完全对齐。

## 4. 推荐架构

推荐采用“三层重组”：

1. 前端应用层：保留 `src/renderer/src` 为主，继续使用 React、HeroUI、Tailwind、Zustand。
2. 桌面适配层：新增统一前端 API 封装层，屏蔽 Electron/Tauri 差异。
3. 桌面宿主层：由 `src-tauri` 提供 command、event、window、tray、updater、sidecar 管理。

建议的新边界如下：

```text
src/
  renderer/
    src/
      api/
        app.ts
        config.ts
        mihomo.ts
        network.ts
        system.ts
      adapters/
        desktop.ts
      ...
src-tauri/
  src/
    main.rs
    commands/
      app.rs
      config.rs
      mihomo.rs
      network.rs
      system.rs
    events/
    state/
    sidecar/
    tray/
    windows/
    updater/
```

核心原则：

1. 前端只调用领域 API，不直接碰 Tauri 原语。
2. Tauri command 按领域组织，不按当前 IPC 文件组织。
3. 能继续由外部进程承担的能力，不急着改写为 Rust 原生实现。

## 5. 能力映射

### 5.1 可以高复用的部分

1. `src/renderer/src` 中的大部分页面、组件、hooks、store。
2. `src/shared` 中的类型定义和部分纯逻辑。
3. Mihomo HTTP API 相关前端展示逻辑。
4. 资源页、日志页、规则页、连接页、统计页的大部分 UI。

### 5.2 需要重写或重组的部分

1. `src/main` 下的大部分 Electron 主进程代码。
2. `src/preload/index.ts` 提供的桥接方式。
3. 依赖 `webContents.send` 的事件分发逻辑。
4. 依赖 `BrowserWindow` 的多窗口管理逻辑。
5. 依赖 `electron-builder` 的构建、签名、更新流程。

### 5.3 建议保留为 sidecar 或外部服务的部分

1. Mihomo 进程管理。
2. 系统代理切换相关服务接口。
3. 需要高权限或平台差异明显的系统操作。

原因很简单：这些能力已经有现成工作路径，短期内没有必要为了“更纯的 Rust 架构”重写全部底层。

## 6. 迁移策略

推荐按以下顺序推进。

### 阶段 0：前端去 Electron 化

目标：让 React 应用不再直接依赖 `window.electron`、`window.api`。

任务：

1. 收敛现有 `src/renderer/src/utils/*-ipc.ts` 调用。
2. 新增 `src/renderer/src/api/*` 作为稳定接口层。
3. 把页面和 store 对 IPC 常量、事件名、桥接对象的直接引用集中移除。
4. 把平台、版本、窗口控制、配置读写等调用改成统一 API。

完成标准：

1. 前端代码中不再出现新的 `window.electron` 直接调用。
2. 前端业务代码只依赖适配层导出的领域方法。

### 阶段 1：搭建 Tauri 宿主和主窗口

目标：让主应用页面在 Tauri 中跑起来。

任务：

1. 初始化 `src-tauri`。
2. 改造前端构建输入，使主页面可由 Tauri 加载。
3. 实现最小 command 集合：
   1. 获取版本。
   2. 获取平台。
   3. 读取应用配置。
   4. 写入应用配置。
   5. 打开目录或文件。
   6. 基础窗口控制。
4. 先确保主窗口单实例可运行。

完成标准：

1. 主窗口能在 Tauri 下正常加载。
2. 基础设置页可读写配置。

### 阶段 2：迁移 Mihomo 能力

目标：恢复 RouteX 的核心运行价值。

任务：

1. 将 `src/main/core/manager.ts` 的职责拆成：
   1. 进程生命周期管理。
   2. 配置生成与校验。
   3. 状态采集与事件广播。
2. 优先保留 Mihomo 作为 sidecar 或外部二进制运行。
3. 用 Tauri event 替代当前基于 `webContents.send` 的推送机制。
4. 先恢复这些能力：
   1. 启动内核。
   2. 停止内核。
   3. 重启内核。
   4. 读取日志。
   5. 读取连接。
   6. 读取流量。
   7. 切换代理组。

完成标准：

1. 主界面可完成核心代理控制。
2. 日志、连接、代理组、规则相关主流程可用。

### 阶段 3：迁移系统能力

目标：恢复桌面客户端必需的环境控制。

任务：

1. 迁移文件读写、保存、导入、打开路径。
2. 迁移系统代理、PAC、DNS、服务状态、权限检查。
3. 迁移快捷键、置顶、窗口显示隐藏、任务栏图标联动。
4. 评估哪些能力使用 Tauri 插件，哪些继续走现有 service。

完成标准：

1. 基础系统代理能力恢复。
2. 设置页中的关键系统操作可执行。

### 阶段 4：迁移托盘和悬浮窗

目标：恢复桌面形态差异化功能。

当前进展（2026-04-10）：

1. Tauri tray 已补齐基础入口和托盘弹窗。
2. 托盘弹窗已补齐主窗口切换、系统代理、TUN、出站模式、悬浮窗、退出应用等基础快捷操作。
3. `triggerMainWindow` 已对齐 Electron 的显示/隐藏切换语义。
4. Tauri 下 `trayIconUpdate` / `updateFloatingWindow` / `updateTrayMenu` 已落到真实宿主通道。
5. 悬浮窗位置持久化已补齐。
6. Windows / Linux 下已补原生托盘菜单代码路径，右键基础菜单项不再继续走 `traymenu` 自定义窗口。
7. 当前原生托盘菜单只覆盖基础项，尚未复刻 Electron 的复杂动态菜单。
8. 代码、构建、打包和应用启动冒烟都已通过；更大范围的 GUI 实机回归留在阶段 6 统一执行。

任务：

1. 用 Tauri tray 重建基础托盘入口。
2. 先实现基础菜单，不先追求当前复杂动态菜单。
3. 单独恢复悬浮窗能力。
4. 再评估是否继续支持自定义托盘窗口和复杂交互。

完成标准：

1. 托盘可显示、点击、切换主窗口。
2. 基础快捷操作可从托盘触发。
3. 悬浮窗可单独显示和关闭。
4. 托盘右键行为已明确收口为“系统原生菜单”或“稳定的自定义窗口”，不能保持未定状态。

### 阶段 5：迁移更新和打包

目标：替换 Electron 发布链路。

当前进展（2026-04-10）：

1. Tauri bundler 已完成打包验证，当前工作区已实际生成 `3.1.0` 的 Windows `msi` / `nsis` 安装产物。
2. Tauri 宿主已接入基于现有 `latest.yml` 的首版更新链路，支持检查更新后的下载、取消、SHA-512 校验和启动安装器。
3. 当前仍未切换到 Tauri 官方 updater / 签名链路；跨平台资产组织、签名和真实安装回归留待后续收口。

任务：

1. 去除 `electron-builder` 依赖链。
2. 配置 Tauri bundler。
3. 评估并接入 Tauri updater。
4. 重新整理 Windows、macOS、Linux 打包产物、签名和发布流程。

完成标准：

1. 可以构建各平台安装包。
2. 自动更新方案至少完成选型和首版验证。

### 阶段 6：回归和收口

目标：保证迁移版本可发布。

任务：

1. 补齐关键路径验证清单。
2. 清理不再需要的 Electron 依赖和目录。
3. 清理过渡期兼容代码。
4. 更新 README 和开发文档。

完成标准：

1. 有稳定的开发、构建、打包路径。
2. Electron 和 Tauri 双栈不再长期并存。

## 7. MVP 范围

首个 Tauri 可用版本建议只覆盖以下能力：

1. 主窗口。
2. 配置读写。
3. 订阅管理主流程。
4. Mihomo 启停与状态读取。
5. 代理组切换。
6. 日志查看。
7. 连接列表。
8. 基础文件导入导出。

MVP 明确不包含：

1. 自定义托盘菜单。
2. 悬浮窗复杂行为。
3. 自动更新。
4. 全量平台细节对齐。
5. 所有设置项一次迁完。

这样做的目的不是缩水，而是先确认 Tauri 宿主层和 Mihomo 主链路成立。

## 8. 关键设计决策

### 8.1 适配层先于功能迁移

如果不先抽离前端适配层，迁移过程会演变成全仓替换 IPC 调用，范围很容易失控。

### 8.2 command 要按领域收敛

不应复制当前 100 多个桌面通道，而应收敛为更少的稳定接口。例如：

1. `app.*`
2. `config.*`
3. `mihomo.*`
4. `system.*`
5. `network.*`

### 8.3 事件流统一收敛

当前很多状态通过窗口直接推送。迁移后应统一为：

1. 后端维护状态。
2. 通过 Tauri event 广播变化。
3. 前端在 providers 或 store 层订阅。

### 8.4 平台高权限操作优先复用现有外部服务

特别是系统代理、DNS、服务控制、权限检查这类平台差异明显的能力，优先复用现有 service 路径比直接重写更稳。

## 9. 主要风险

### 9.1 Mihomo 进程管理

风险最高。因为它涉及：

1. 外部进程生命周期。
2. 配置生成与校验。
3. 异常退出重试。
4. 日志监听。
5. 平台权限差异。

### 9.2 系统代理与权限

Windows、macOS、Linux 的行为差异大。迁移时最容易出现“应用能跑，但环境控制失效”的问题。

### 9.3 多窗口与托盘

Tauri 支持多窗口和托盘，但实现方式与 Electron 不同。当前自定义托盘窗口和悬浮窗不应在第一阶段强行复刻。

### 9.4 自动更新

Electron 当前更新逻辑是显式下载和校验安装包。迁移后不能默认认为 Tauri updater 能无缝接手全部现状。

## 10. 验证计划

每个阶段都要有明确验证，不接受“能编译就算完成”。

### 10.1 阶段级验证

1. 阶段 0：确认前端代码不再依赖 Electron preload。
2. 阶段 1：确认主窗口启动、配置读写、平台信息获取正常。
3. 阶段 2：确认 Mihomo 启停、日志、连接、代理组切换正常。
4. 阶段 3：确认系统代理关键路径可执行。
5. 阶段 4：确认托盘与悬浮窗基础能力可用。
6. 阶段 5：确认至少一个平台能完整打包安装。

### 10.2 发布前验证

1. Windows 主路径验证。
2. macOS 权限和系统代理验证。
3. Linux 托盘与窗口行为验证。
4. 异常退出和重启恢复验证。
5. 配置迁移和兼容性验证。

## 11. 建议的首批落地任务

如果按执行顺序拆成具体任务，建议先做下面 8 项：

1. 新建 `docs/tauri-rewrite-plan.md`，作为迁移基线文档。
2. 盘点前端所有 `window.electron`、`window.api` 调用点。
3. 新建前端 `api` 目录并定义统一接口。
4. 先把版本、平台、配置读写迁到统一接口。
5. 初始化 `src-tauri` 并跑通主窗口。
6. 接入最小 command 集合。
7. 迁移 Mihomo 启停和基础状态查询。
8. 补一份阶段验收清单。

## 12. 不做事项

为了控制范围，以下事项在当前方案中明确不做：

1. 不做 Electron 到 Tauri 的机械式 API 对照表。
2. 不做一次性全功能对齐承诺。
3. 不做未验证前提下的 Rust 全量重写。
4. 不做 UI 改版。

## 13. 退出条件

当以下条件同时满足时，可以认为 Tauri 重写达到“可替代 Electron 主版本”的门槛：

1. 主窗口、配置、订阅、Mihomo 核心链路稳定。
2. 系统代理关键路径在目标平台可用。
3. 托盘或等效入口已恢复。
4. 构建、打包、发布流程已成型。
5. Electron 仅保留为回退分支，不再继续并行开发。
